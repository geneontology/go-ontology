What is a Claude skill?

A skill is a folder of instructions and optional resources that teaches Claude how to do a specialized task well, your way.

  • Lives in .claude/skills/<name>/
  • Always contains a SKILL.md
  • May bundle scripts, templates, reference docs
  • Loaded on demand — only when relevant

Think of it as an expert checklist Claude pulls off the shelf at the right moment, instead of cramming every procedure into one giant prompt.

.claude/
└── skills/
    └── term-obsoletion/
        └── SKILL.md      ← the instructions
        (+ scripts, refs,
           templates …)

portableversion-controlledcomposable

The problem skills solve

Without skills

  • One enormous system prompt with every rule for every task
  • Wastes context on procedures you aren't using right now
  • Hard to maintain, hard to share, easy to contradict

With skills

  • Each procedure is a small, focused file
  • Claude reads only the metadata until a skill is needed
  • Full instructions load just in time
  • Reusable across sessions, repos, and teammates
Core idea: progressive disclosure — show Claude a little, then more only when it matters.

Anatomy of a SKILL.md

---
name: term-obsoletion
description: For handling the
  process around obsoleting GO
  terms. ... use this skill ...
---

...detailed procedure...

Two parts

1. YAML frontmatter

  • name — the skill's id (and its /slash command)
  • description — when to use it. This is the most important line you write.

2. The body (Markdown)

  • How to do the task: steps, rules, examples, gotchas
Claude sees only name + description by default. It reads the body when it decides the skill applies.

How Claude discovers & invokes a skill

read all skill descriptions→ task matches a description→ load full SKILL.md→ follow it (+ open bundled files)

Two ways a skill fires

  • Automatic — Claude recognizes the task fits a description and pulls the skill in itself.
  • Explicit — the user (or another instruction) types /term-obsoletion to invoke it by name.

Why description is king

It's the only thing Claude sees up front. A vague description = the skill never fires. A sharp one says exactly which situations it covers.

description: For working with any
  terms in the catalytic activity
  branch; these often correspond
  to RHEA or EC terms

Progressive disclosure: 3 levels

Level 1

Metadata

name + description for every skill, always loaded. Tiny. Lets Claude decide what's relevant.

Level 2

SKILL.md body

Loaded when the skill is triggered. The full procedure, examples, and rules for the task at hand.

Level 3

Bundled resources

Scripts, templates, large reference docs — opened only if the body points to them. Keeps everything else lean.

The result: deep expertise available on demand, without paying the context cost when you don't need it.

Authoring a good skill

Do

  • Write a trigger-focused description ("use this when…")
  • Keep the body actionable: steps, checklists, concrete examples
  • Show real input/output (exemplar terms work great)
  • Name the exact tools/commands to run
  • Call out anti-patterns and gotchas

Avoid

  • Generic descriptions ("helps with ontology stuff")
  • Dumping everything in one giant body — push detail to Level 3
  • Guessing IDs / refs instead of pointing to a lookup procedure
  • Overlapping skills that fight over the same trigger
A skill is documentation that Claude actually executes. Write it for a competent newcomer who will follow it literally.

Next: a closer look at each.

/research

Background research from literature and the web, captured in a RESEARCH.md file.

What it does

  • Web / literature search; prefers recent reviews + PMIDs
  • Supports a deep-research-client for focused deep dives
  • Writes findings to RESEARCH.md (not committed)

Key rules

  • GO is pan-species — don't over-fit to human biology
  • Skip it for trivial typo / compositional fixes (mark N/A)
  • Post the report back on the issue — the temp file vanishes
Used to satisfy the RESEARCH and REFERENCE-VALIDATION steps of the editing checklist — every PMID must be verified, never fabricated.

# produces → RESEARCH.md

/design-pattern

Ensures labels, text definitions, and logical definitions stay consistent within and across terms.

  • Documents relevant patterns in DESIGN_PATTERNS.md
  • Patterns live as DOSDP YAML in src/patterns/*.yaml
  • Required before adding any intersection_of axioms
  • Cross-checks against real precedent via obo-grep.pl
id: GO:0001710
name: mesodermal cell fate commitment
intersection_of: GO:0045165 ! cell
  fate commitment
intersection_of: results_in_commitment_to
  CL:0000222 ! mesodermal cell

Genus + differentia: the logical def mirrors the label and text def.

/external-term-lookup

Never guess an ID. Confirm every external term against the source ontology.

In the import closure

obo-grep.pl --noheader -r cysteine \
  src/ontology/imports/chebi_import.obo

Anything else (via OAK + OLS)

runoak -i ols:chebi search cysteine

Query GO structure locally

runoak -i simpleobo:src/ontology/\
go-edit.obo tree -p i GO:0160022
The backbone for finding CHEBI / CL / UBERON / NCBITaxon IDs used in definitions and relationships.

/chemical-entity

CHEBI is full of traps — the friendliest-looking label is often the wrong term.

The golden rules

  • pH 7.3 canonicalization: use the major species at physiological pH
  • Pick the correct L / D stereoisomer
  • Verify parent classes & cross-references

Mandatory for any term touching a chemical or CHEBI ID.

id: GO:0005290
name: L-histidine transmembrane
  transporter activity
def: "...L-histidine is 2-amino-3-
  (1H-imidazol-4-yl)propanoic acid."
Charged forms (e.g. L-histidinium) vs neutral matter for correct logical definitions.

/reaction

For the catalytic-activity branch — terms that map to RHEA or EC.

  • One source of truth reaction in the def xref (usually RHEA)
  • Mappings qualified with SKOS via {source="…"}
  • EC numbers often appear as RELATED synonyms
def: "Catalysis of the reaction:
  ... = ..." [RHEA:17341]
xref: EC:1.1.1.101
  {source="skos:exactMatch"}
xref: RHEA:36175
  {source="skos:narrowMatch"}

/mapping

Cross-references (xref) carry meaning — say how the terms relate.

  • Mappings are term-level, not the same as def provenance
  • Modern mappings include a SKOS predicate
  • Don't confuse mappings with external logical definitions
predicatemeaning
exactMatchsame concept
narrowMatchexternal is narrower
broadMatchexternal is broader
id: GO:0000031
name: mannosylphosphate
  transferase activity
xref: MetaCyc:RXN-22693
  {source="skos:narrowMatch"}

Check the mapped term, and look at mappings holistically.

/taxon-constraint

Restrict where a term may be used — in separate TSV files, not the OBO.

  • Files: src/taxon_constraints/never_in_taxon.tsv & only_in_taxon.tsv
  • Favor parsimony — constrain to a more general clade when sensible
  • Don't infer constraints from a GO name alone
  • Constraints on referenced CL/UBERON terms are inherited — no duplication
# query NCBITaxon via OAK
runoak -i ols:ncbitaxon \
  info Mammalia
Great care required: "heart development" may be meant broadly (incl. insect hearts) or narrowly.

/term-obsoletion

The most involved skill — safely retiring a term and cleaning up after it.

Three flavors

  1. With direct replaced_by
  2. With consider candidates
  3. With no replacement

No more merges/alt_ids — a "merge" request means obsoletion-with-replacement.

An obsolete term must

  • drop all is_a / relationship / intersection_of
  • be renamed obsolete <name>
  • prefix def with OBSOLETE.
  • carry a tracker item + reason in a comment
  • have all inbound references rewired first

How skills plug into the GO workflow

The repo's CLAUDE.md defines a PLAN checklist; skills are the specialists it calls.

PLAN→ /research→ /external-term-lookup→ /design-pattern→ EDIT→ validate

Specialized edits route to skills

  • chemicals → /chemical-entity
  • RHEA / EC → /reaction
  • obsoletion → /term-obsoletion
  • taxon scope → /taxon-constraint
  • xrefs → /mapping
Skills keep each procedure correct and consistent, so the main guide stays a high-level map rather than a 2000-line manual.

Write your own skill

  1. Create .claude/skills/<name>/SKILL.md
  2. Add frontmatter — a sharp name and trigger-focused description
  3. Write the procedure: steps, exemplars, commands, anti-patterns
  4. Push large refs/scripts into sibling files (Level 3)
  5. Commit it — now it travels with the repo for everyone
